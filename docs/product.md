# 产品设计

> 通过自动化执行团队既有代码规范，将低价值、重复的规范审查从人工 Code Review 中剥离出来，实现更低成本、更高一致性的工程质量保障。

## 1. 背景

在工程团队的 Code Review 中，大量审查精力被消耗在**团队已达成共识的私有代码规范**检查上。

这类规范通常稳定且明确，但依旧依赖人工检查，带来了以下问题：

- 审查成本随团队规模和代码量增长而快速上升
- 审查结果依赖个人经验，一致性难以保证
- 高级工程师时间被低价值、重复性的规范检查占用

可以通过工程化手段，将这部分稳定、可重复的审查工作自动化执行。

---

## 2. 需求

> **优先保证错误检出率（Recall），而非追求零误报。**

核心需求包括：

- 降低人工规范审查的成本
- 提升规范执行的一致性与覆盖度
- 使人工 Code Review 聚焦于设计、架构和复杂语义问题

---

## 3. 产品定位

> 一个以规则为核心的自动化规范执行层，用于一致、可重复地执行团队已达成共识的代码规范。

能做的：

- 仅执行团队已明确约定的规范
- 不引入个人偏好或推测性的“最佳实践”
- 设计与架构决策始终由人工审查负责

不做的：

- 判断代码是否“优雅”或符合个人风格偏好
- 在无明确共识的情况下引入新规范
- 自动批准或拒绝代码变更
- 取代人工进行架构或设计层面的讨论
